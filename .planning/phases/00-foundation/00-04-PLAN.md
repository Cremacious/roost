<task type="checkpoint:human-action">
  <name>Task 2: Generate baseline migration and register it</name>

  <action>
This task establishes the Drizzle migration baseline using the community-established pattern (RESEARCH.md Pattern 3). Each step is described precisely. Read all steps before executing.

**Step 1: Confirm drizzle.config.ts is correct**

```bash
cat apps/web/drizzle.config.ts
```

Must show `out: './migrations'` and `process.env.DEV_DATABASE_URL!`. If it shows `DATABASE_URL` or `out: './drizzle'`, stop and fix Plan 01 output first.

**Step 2: Generate the baseline migration file**

From the repo root (the delegate script handles the workspace path):
```bash
npm run db:generate -- --name=initial
```

This creates `apps/web/migrations/0000_initial.sql`. The file will contain CREATE TABLE statements for every table in the schema. This is normal — the tables already exist on the v2-dev branch (copied from production).

Verify the file exists:
```bash
ls apps/web/migrations/
# Expected: 0000_initial.sql and a meta/ directory
```

**Step 3: Comment out ALL SQL in 0000_initial.sql**

Open `apps/web/migrations/0000_initial.sql` in your editor.

The file looks like:
```sql
CREATE TABLE "users" (
  ...
);
CREATE TABLE "households" (
  ...
);
-- etc.
```

Wrap the ENTIRE SQL body in a block comment, and add a header comment:
```sql
-- BASELINE: schema already applied via db:push on production.
-- This file registers the initial schema in __drizzle_migrations so
-- drizzle-kit treats the current schema as "already applied".
-- The SQL below is documentation and rollback reference only.
-- DO NOT run this file directly against any database.
/*
[paste all the original SQL here — CREATE TABLE statements, indexes, etc.]
*/
```

Every line of SQL must be inside the `/* ... */` block. The file should have NO executable SQL outside of comments when you are done.

**IMPORTANT: Do not run the next step until you have verified every SQL statement is commented out.**

**Step 4: Run drizzle-kit migrate to register the baseline**

```bash
npm run db:migrate
```

This command:
1. Creates the `__drizzle_migrations` table on the v2-dev branch (if it doesn't exist)
2. Inserts a row recording that `0000_initial.sql` has been applied

Because the SQL is commented out, no DDL is executed. The command succeeds because the schema state is unchanged.

Expected output: something like `[✓] Done! Applied 1 migration(s)` or `Migration applied successfully`.

If you see `relation "users" already exists` — you forgot to comment out the SQL. Stop, comment out the SQL, and run `npm run db:migrate` again.

**Step 5: Restore 0000_initial.sql to full DDL**

Remove the `/* */` block comment wrapper, restoring all CREATE TABLE statements to their original form. Keep the header comment at the top:

```sql
-- BASELINE: schema already applied via db:push on production.
-- This file is documentation and rollback reference.
-- It was registered in __drizzle_migrations as a no-op baseline on [date].
CREATE TABLE "users" (
  ...
);
-- etc.
```

**Step 6: Verify migration is registered**

Run generate again — it should produce zero new migration content:
```bash
npm run db:generate -- --name=check_clean
```

This creates a new file (e.g., `0001_check_clean.sql`) that should be EMPTY or contain only a single comment. If it has SQL content, the schema is out of sync. Delete the `0001_check_clean.sql` file after verifying.

Also verify via SQL:
```bash
cd apps/web
npx tsx -e "
const { neon } = require('@neondatabase/serverless');
require('dotenv').config({ path: '.env.local' });
const sql = neon(process.env.DEV_DATABASE_URL);
sql('SELECT * FROM __drizzle_migrations').then(r => {
  console.log('Registered migrations:', r.length);
  console.log(r);
});
"
```

Expected: 1 row with hash matching `0000_initial.sql`.

**Step 7: Delete the check_clean migration file**

```bash
rm apps/web/migrations/0001_check_clean.sql 2>/dev/null || true
```

**Step 8: Commit the migration files**

```bash
git add apps/web/migrations/ apps/web/.env.example
# Do NOT add apps/web/.env.local — it contains secrets and must be gitignored
git commit -m "feat(foundation): establish Drizzle migration baseline on v2-dev (FOUN-03, MIG-01)"
```

Verify `.env.local` is in `.gitignore`:
```bash
grep "\.env\.local" apps/web/.gitignore || grep "\.env\.local" .gitignore
```

If neither has it, add `apps/web/.env.local` to the root `.gitignore`.
  </action>

  <verify>
    <automated>

grep -c "CREATE TABLE" apps/web/migrations/0000_initial.sql

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-11 | Tampering | drizzle.config.ts credential target | mitigate | Config reads `process.env.DEV_DATABASE_URL!` with no fallback. | T-00-12 | Tampering | Running migrate with live SQL | mitigate | Step 3 (comment out SQL) + developer checkpoint before Step 4 prevents accidental DDL execution. The checkpoint:human-action gate enforces human review. |
| T-00-13 | Information Disclosure | apps/web/.env.local | mitigate | .env.local is gitignored (verify in Step 8). DEV_DATABASE_URL connection string must never be committed. `.env.example` documents variable names without values. |
| T-00-14 | Tampering | Accidental production DB access | mitigate | V2 codebase (`apps/web/`) never contains `DATABASE_URL`. The production connection string is only in the root `.env.local` which is used by V1 on master. V2's `apps/web/.env.local` contains only DEV_DATABASE_URL. The Neon dev branch is isolated — any DDL mistakes cannot affect production data. |
</threat_model>
